How it works

Certificate Transparency for AI

The same model that secures the web's TLS certificates (RFC 6962), adapted for compute receipts. No consensus. No tokens. Just append-only logs and cross-witnessing.

🔒

Receipt

21 fields: model, input hash, output hash, proof, provider signature. SHA-256 of canonical bytes = receipt ID. Tamper one bit, the hash breaks.

🌳

Merkle log

Transfers appended to independent transparency logs. Domain-separated hashing. Every entry has an inclusion proof. Double-spends are impossible.

👁

Witnessing

3 log servers in different countries countersign each other's checkpoints. 2-of-3 threshold. Rollback = instant detection. No single point of trust.

For developers

Query the network directly

Every receipt, proof, and checkpoint is accessible via HTTP. No auth, no API keys.

# Health — is the log running?
curl https://log1.tesseravcr.org/v1/health

# Checkpoint — latest signed Merkle root
curl https://log1.tesseravcr.org/v1/checkpoint

# Entry — a specific receipt with provenance
curl https://log1.tesseravcr.org/v1/entry/28

# Proof — Merkle inclusion proof for any entry
curl https://log1.tesseravcr.org/v1/proof/28
